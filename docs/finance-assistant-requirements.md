# 个人财务助手 - 需求文档

## 1. 产品概述

一个基于 LLM 的个人财务助手，支持自然语言交互，自动记录收支、管理预算、分析消费趋势，帮助用户掌握个人财务状况。

---

## 2. 核心架构：三层记忆

| 层级 | 存储介质 | 数据内容 | 生命周期 |
|------|---------|---------|---------|
| **短期记忆** | 内存 / Redis | 当前对话的最近 N 轮上下文 | 会话级 |
| **结构化记忆** | SQLite | 收支记录、预算、分类、月度汇总 | 永久 |
| **长记忆** | mem0ai | 用户偏好、消费习惯、财务目标、反馈 | 长期演进 |

### 2.1 结构化数据表

```
expenses          支出记录（user_id、金额、分类、描述、日期、支付方式、category_l1_id、category_l2_id、category_confidence、needs_review）
incomes           收入记录（user_id、金额、来源、描述、日期、category_l1_id、category_l2_id、category_confidence、needs_review）
categories        分类体系（code、name、direction、level、parent_id、is_active）
category_aliases  分类别名（category_id、alias、locale、weight）
budgets           预算定义（user_id、category_id、金额、周期类型、起止时间、预警阈值、is_active）
monthly_summaries 月度汇总（user_id、年、月、总支出、总收入、Top 分类）
```

### 2.1.1 数据完整性约束

- 所有表必须包含 `user_id` 字段实现数据隔离
- 金额字段使用 `DECIMAL(12,2)`，精度 2 位小数
- 支出金额 > 0，收入金额 > 0，负金额视为冲正/撤销操作
- 单笔最大金额限制：9,999,999.99
- 所有时间字段使用 UTC 存储，读取时转换为 `Asia/Shanghai`

### 2.1.2 去重与幂等性

- 同一次会话中，相同金额 + 相同分类 + 相同描述 + 相同日期的记录视为潜在重复
- 潜在重复应触发确认提示，而非直接创建新记录
- 异步写入需保证幂等，相同 `trace_id` + 相同数据只落库一次

### 2.1.3 数据修正与撤销

- 用户可通过自然语言修改已有记录（"午餐其实是 35 不是 30"）
- 用户可删除已有记录
- 所有修正/删除操作保留审计日志（soft delete 或 audit_trail 表）

### 2.1.4 月度汇总计算策略

- `monthly_summaries` 采用写时更新策略：每次写入 expenses/incomes 时增量更新对应月份汇总
- 异常情况下支持重新计算（recompute 接口）

### 2.2 记忆存储决策

| 信息类型 | 示例 | 存储位置 |
|---------|------|---------|
| 具体收支 | 「今天打车花了 45」 | SQLite |
| 用户偏好 | 「咖啡单独分一类」 | mem0ai |
| 财务目标 | 「年底交通预算控制在 5000 以内」 | mem0ai |
| 习惯模式 | 「周末外卖明显增多」 | mem0ai |
| 分类纠正 | 「午餐不算餐饮，算外卖」 | mem0ai (Feedback) |
| 查询讨论 | 「帮我看看这个月超支没」 | 短期记忆 |

## 3. 核心功能

### 3.1 意图识别

用户输入通过 LLM 轻量级分类，拆解为独立意图：

```json
{
  "store_intents": [
    {"type": "structured", "data": {"amount": 300, "category": "餐饮", "description": "吃饭"}},
    {"type": "long_term", "data": {"type": "goal", "content": "每周餐饮预算 500"}}
  ],
  "query_intent": {
    "type": "structured",
    "params": {"date_range": "last_week", "category": "餐饮", "operation": "sum"}
  }
}
```

**关键设计**：一条输入可同时包含存储意图和查询意图，两者独立处理。

### 3.2 存储（异步）

- **结构化数据**：写入 SQLite，自动关联分类、日期
- **长记忆**：写入 mem0ai，更新用户偏好/目标/习惯
- **后台执行**：不阻塞用户回复，用户无感知延迟

### 3.3 查询（同步）

- **结构化查询**：SQL 聚合（SUM/GROUP BY/时间范围）
- **长记忆查询**：mem0ai 语义检索（偏好、目标、习惯）
- **混合查询**：同时读取 SQLite + mem0ai，LLM 综合生成回复

### 3.4 回复生成

LLM 结合查询结果、用户画像、对话上下文生成自然语言回复。

---

## 4. 短期记忆压缩

当对话轮次超过阈值（默认 20 轮）时触发：

### 4.1 压缩策略

```
1. 提取关键信息 -> 写入 SQLite / mem0ai
2. 生成对话摘要 -> 注入系统提示
3. 清空旧消息 -> 保留最近 N 条
```

### 4.2 压缩触发

- 消息数 > `summary_threshold`（默认 30）时触发
- 压缩后保留最近 `max_turns`（默认 20）条
- 摘要追加到系统提示，避免信息丢失

## 5. 系统流程

```
用户输入
  |
  v
[意图识别 - LLM 轻量调用]
  |
  v
[意图路由器]
  |
  ├──── 存储意图（异步，用户无感知）────┐
  |                                     |
  v                                     v
SQLite 写入                        mem0ai 写入
(结构化数据)                        (长记忆)

  |
  ├──── 查询意图（同步，主回复路径）────┐
  |                                     |
  v                                     v
SQL 查询                            mem0ai 检索
(收支/预算/汇总)                     (偏好/目标/习惯)
  |                                     |
  └──────────────┬──────────────────────┘
                 v
         [回复生成 - LLM 主对话]
                 |
                 v
              用户回复
```

## 7. 技术选型建议

| 组件 | 方案 | 说明 |
|------|------|------|
| 主框架 | LangGraph | 状态机编排意图路由 |
| 结构化存储 | SQLite + SQLModel | 轻量、零部署、已有依赖 |
| 长记忆 | mem0ai | 已有依赖，支持语义检索 |
| 意图识别 | 小模型（GPT-4o-mini / Qwen-Turbo） | 降低成本和延迟 |
| 主对话 | Claude / GPT-4o | 高质量回复生成 |
| 对话缓存 | 内存（单用户） | 暂不需要 Redis |

## 9. 关键设计原则

1. **存储不是查询的前提** — 用户同时说和问时，先回答再存储
2. **一次调用完成提取** — 不分别调多次 LLM
3. **小模型做路由** — 意图识别不需要强推理能力
4. **结构化用表，定性用记忆** — 需要聚合的进 SQLite，描述用户的进 mem0
5. **用户无感知延迟** — 异步存储，同步回复

## 10. 安全与成本控制

### 10.1 接口安全

- `/v1/chat` 需要 API Key 认证（Bearer Token）
- 输入长度限制：单条消息最大 2000 字符
- 频率限制：单用户每分钟最多 20 次请求
- SQL 查询路径必须使用参数化查询，禁止字符串拼接

### 10.2 LLM 成本控制

- 意图识别使用小模型（GPT-4o-mini / Qwen-Turbo），单次调用 token 上限 500
- 主对话使用高质量模型（Claude / GPT-4o），单次调用 token 上限 2000
- 设置月度 LLM 费用上限，达到阈值后自动降级到更便宜的模型
- 短期记忆压缩时优先使用更便宜模型生成摘要

### 10.3 数据隐私

- 日志中不记录原始金额和个人信息（脱敏处理）
- mem0 数据保留期限：用户可配置，支持一键删除所有长期记忆（Right to be Forgotten）
- LLM API 调用会传输财务数据到第三方，需在配置中明确声明所用提供商的隐私政策
- 所有用户请求生成 `trace_id`，用于审计和问题追踪

## 11. 错误处理与降级

### 11.1 LLM 提供商故障

- 超时阈值：意图识别 10s，主对话 15s
- 重试策略：最多 2 次，指数退避（1s, 2s）
- 降级策略：LLM 不可用时返回模板回复（"服务暂时不可用，请稍后再试"），但异步存储路径仍继续执行

### 11.2 mem0 不可用

- mem0 连接失败时不阻塞主流程
- 降级为仅结构化查询回答模式
- 记录警告日志，后台持续重试
- mem0 恢复后自动补齐缺失的长记忆写入

### 11.3 SQLite 并发与锁

- 启用 WAL 模式（`PRAGMA journal_mode=WAL`）
- 写入超时：`PRAGMA busy_timeout=5000`（5秒）
- FastAPI 异步上下文使用 `check_same_thread=False`
- 写入冲突重试：最多 3 次，每次间隔 500ms

### 11.4 数据库备份

- 每日自动备份 SQLite 数据库文件
- 备份文件保留最近 7 天
- 备份路径可配置

### 11.5 原子性保证

- 存储操作要么全部成功，要么全部回滚
- 异步任务失败写入 `failed_operations` 表（死信队列）
- 死信队列支持手动重试

## 12. 可观测性

### 12.1 日志

- 结构化日志（JSON 格式），支持按模块设置日志级别
- `trace_id` 贯穿全链路（请求 → 意图识别 → 查询 → 回复 → 异步存储）
- 日志级别：API 层 INFO，图编排 DEBUG，服务层 INFO，数据库 WARNING

### 12.2 健康检查

- `GET /health` — 返回服务状态（SQLite、LLM、mem0 的连通性）
- `GET /ready` — 启动就绪检查（所有依赖可用）
- 健康检查不记录业务日志

### 12.3 指标

- 请求延迟 P50/P95/P99
- LLM 调用次数与 token 消耗
- 异步存储成功率 / 失败率
- 分类准确率（自动统计 needs_review 比例）

## 13. 时间语义规范

所有时间表达统一按 `Asia/Shanghai`（UTC+08:00）解析：

| 表达 | 含义 |
|------|------|
| 今天 | 当日 00:00 - 23:59 |
| 昨天 | 前一日 00:00 - 23:59 |
| 这周 | 本周一 00:00 - 本周日 23:59 |
| 上周 | 上周一 00:00 - 上周日 23:59 |
| 这个月 | 本月 1 日 00:00 - 月末 23:59 |
| 上个月 | 上月 1 日 00:00 - 上月末日 23:59 |
| 最近 7 天 | 今天往前推 7 天（含今天） |
| 最近 30 天 | 今天往前推 30 天（含今天） |

## 14. 分类与枚举细则（Phase 1 必做）

### 10.1 分类体系定义

- 分类不使用代码硬编码 Enum，统一以数据库字典表维护，支持在线扩展。
- 分类支持二级结构：
  - 一级分类（L1）：用于统计和预算主维度。
  - 二级分类（L2）：用于精细分析和个性化。
- 每个分类必须绑定 `direction`：
  - `expense`（支出）
  - `income`（收入）
- 允许仅命中一级分类后落库（二级可为空），不阻塞记账。

### 10.2 基础分类建议（初始化）

- 支出一级：餐饮、交通、居住、购物、娱乐、医疗、教育、通讯、其他
- 收入一级：工资、奖金、报销、理财、转账、退款、其他
- 示例二级：
  - 餐饮 -> 早餐/午餐/晚餐/咖啡/外卖
  - 交通 -> 地铁/公交/打车/加油
  - 工资 -> 基本工资/绩效/补贴
  - 理财 -> 利息/基金收益

### 10.3 分类匹配策略（LLM + 规则）

- 步骤 1：LLM 抽取候选字段：`direction`、`raw_category`、`amount`、`description`
- 步骤 2：分类标准化服务按以下顺序匹配：
  1. 分类别名精确匹配
  2. 语义相似度匹配
  3. 规则兜底（关键词映射）
- 输出统一包含：`matched_category_id`、`match_type`、`confidence`

### 10.4 低置信度与未命中处理

- 置信度阈值建议：
  - `confidence >= 0.85`：自动分类并落库
  - `0.60 <= confidence < 0.85`：落一级分类，标记 `needs_review=true`
  - `confidence < 0.60`：落 `待分类/其他`，并在回复中发起一次确认
- 匹配失败时不丢账：先落库到 `待分类`，后续可纠正。
- 用户纠正后：
  - 更新 `category_aliases`
  - 写入 mem0 反馈记忆
  - 后续同类输入优先采用该偏好

### 10.5 数据模型补充

新增/补充以下结构化表字段：

```text
categories(
  id, code, name, direction, level, parent_id,
  is_active, created_at, updated_at
)

category_aliases(
  id, category_id, alias, locale, weight,
  created_at, updated_at
)

expenses / incomes 补充：
- category_l1_id
- category_l2_id (nullable)
- category_confidence
- needs_review
```

### 10.6 验收标准补充

- 分类方向识别准确率（收入/支出）>= 98%
- 一级分类命中率 >= 95%
- 二级分类命中率 >= 85%（在含明确信号样本中）
- 低置信度样本必须进入 `needs_review` 或 `待分类`，不得静默误分类
- 用户纠正后，同类表达在后续 5 次内命中率显著提升（需有回归测试证明）
