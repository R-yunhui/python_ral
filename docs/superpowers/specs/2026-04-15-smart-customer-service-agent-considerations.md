# 智能客服 Agent 系统 — 全景考量

> 目标：设计一个生产级、可直接对外服务的智能客服 Agent 系统
> 首发渠道：Web 聊天窗口
> 日期：2026-04-15

---

## 一、核心架构层

### 1.1 Agent 编排引擎

- **对话状态机**：管理用户对话的生命周期（空闲→交互中→等待工具→等待人工→结束）
- **意图识别路由**：用户消息进来后，先分类（咨询/投诉/查询/闲聊/转人工），再决定走哪个子流程
- **多轮对话管理**：上下文追踪、slot filling（槽位填充）、追问机制
- **Agent 组合模式**：单 Agent（简单）、Multi-Agent（分工协作：意图识别 Agent + 知识检索 Agent + 工单 Agent + 情感安抚 Agent）
- **回退策略**：LLM 挂了/超时/输出不符合预期时的降级方案

### 1.2 工具/能力系统

- **工具注册与发现**：Agent 能调用的外部能力（查订单、查库存、改地址、发优惠券、创建工单）
- **工具执行沙箱**：工具调用的权限控制、重试机制、超时控制、幂等性
- **工具组合**：多个工具的组合调用（先查订单状态 → 再查物流 → 再发补偿券）
- **动态工具注入**：根据意图只给 Agent 相关的工具，减少 prompt 膨胀和幻觉

### 1.3 知识系统

- **知识库检索（RAG）**：产品文档、FAQ、政策条款、操作手册的向量化检索
- **知识版本管理**：知识更新后旧版本的失效时间线
- **知识命中率监控**：哪些知识常被问到、哪些找不到、哪些回答被用户标记为无用
- **实时知识更新**：运营人员可随时更新知识，无需重新部署

## 二、对话质量层

### 2.1 输出控制

- **系统提示工程**：角色定义、行为边界、语气风格、安全红线
- **输出格式约束**：结构化输出（JSON schema）、富文本（Markdown/卡片）
- **敏感词过滤**：输出前扫描，避免违规内容
- **Hallucination 防护**：回答必须基于知识库，不知道就说不知道
- **多语言支持**：中文为主，可扩展英文等

### 2.2 个性化与记忆

- **用户画像**：姓名、等级、历史行为、偏好标签
- **短期记忆**：当前会话的上下文（窗口大小、摘要策略）
- **长期记忆**：跨会话的用户信息沉淀（上次投诉了什么、常问什么问题）
- **会话续接**：用户隔几天回来，Agent 能记得之前的对话

### 2.3 情感与体验

- **情感识别**：检测用户愤怒/焦虑/满意，调整回复策略
- **安抚话术**：检测到负面情绪时的自动安抚和升级
- **人格一致性**：始终保持统一的客服人设（专业、友善、高效）
- **响应时间感知**：长任务时的"正在为您查询..."过渡语

---

## 三、人机协作层

### 3.1 人工接管

- **转人工触发条件**：用户主动要求 / 情感识别到愤怒 / 置信度低 / 超出知识范围
- **无缝切换**：Agent → 人工的平滑过渡，不丢上下文
- **人工辅助模式**：Agent 给人工客服推荐回复，人工确认后发出
- **回退给 Agent**：人工处理完后，可以再交还给 Agent

### 3.2 工单系统

- **自动建单**：Agent 无法当场解决的问题自动生成工单
- **工单状态追踪**：用户可以问"我的工单处理到哪了"
- **工单分类与路由**：自动分类给对应的处理部门

### 3.3 运营后台

- **对话监控面板**：实时看当前有多少会话、Agent 表现如何
- **对话记录回溯**：搜索、过滤、回放历史对话
- **知识管理**：上传文档、编辑 FAQ、设置同义词
- **数据看板**：解决率、满意度、转人工率、平均响应时间

---

## 四、工程可靠性层

### 4.1 性能

- **首字延迟**：流式输出，TTFB < 1s
- **并发能力**：支持多少并发会话
- **缓存策略**：相似问题的回答缓存、知识向量缓存
- **模型路由**：简单问题走便宜/快的模型，复杂问题走强的模型

### 4.2 可用性与容错

- **LLM 降级**：主模型挂了 → 备用模型 → 规则引擎 → 默认回复
- **幂等性**：网络重试不会导致重复操作（如下两次单）
- **限流**：防止用户刷消息导致 LLM 费用爆炸
- **健康检查**：各组件的存活探测

### 4.3 可观测性

- **全链路追踪**：一次对话的完整链路（用户消息 → 意图识别 → 工具调用 → LLM 生成 → 返回）
- **日志结构化**：每步的输入输出、耗时、token 消耗
- **指标采集**：延迟、错误率、token 用量、工具调用次数
- **告警**：错误率突增、LLM 超时频繁、token 费用异常

---

## 五、安全与合规层

### 5.1 数据安全

- **用户数据加密**：敏感信息（手机号、地址）的存储加密
- **数据脱敏**：日志和监控中的 PII 脱敏
- **数据保留策略**：对话记录保留多久、如何清理
- **权限隔离**：不同角色的数据访问权限

### 5.2 内容安全

- **输入审查**：用户消息的有害内容检测
- **输出审查**：Agent 回复的安全审核
- **合规红线**：不做医疗建议、法律意见等越界行为
- **审计日志**：谁在什么时候做了什么，不可篡改

### 5.3 成本控制

- **Token 预算管理**：单次对话的 token 上限、每日总预算
- **成本分摊**：按渠道/业务线/用户统计 LLM 费用
- **模型选择策略**：能用 flash 就不用 pro，能用缓存就不用新请求

## 六、扩展性层

### 6.1 多渠道接入

- **渠道适配层**：Web、微信、企业微信、钉钉、电话的统一抽象
- **渠道特性**：每个渠道的消息格式、富文本能力、用户 ID 体系不同

### 6.2 多租户

- **企业隔离**：不同企业的数据、知识库、配置完全隔离
- **企业定制**：每家企业的 Agent 人设、知识库、工具可以不同
- **共享基础设施**：同一套代码服务多个租户

### 6.3 插件生态

- **自定义工具**：企业可以自己注册工具供 Agent 调用
- **自定义流程**：可配置的对话流程（欢迎语 → 引导 → 分流）
- **Webhook**：外部系统回调通知

---

## 七、迭代优化层

### 7.1 评估与测试

- **自动化评估**：回答质量打分（基于参考答案或 LLM-as-judge）
- **A/B 测试**：不同 prompt/模型/策略的对比实验
- **回归测试**：知识更新后，之前能回答的问题是否还能回答
- **红队测试**：故意输入攻击性/越狱提示，测试安全边界

### 7.2 持续学习

- **未解决问题收集**：Agent 答不上来的问题自动归档
- **知识缺口发现**：高频未解决问题 → 提示补充知识
- **用户反馈闭环**：点赞/差评 → 分析原因 → 改进 prompt 或知识
- **对话数据分析**：聚类分析用户最常问的问题类型
