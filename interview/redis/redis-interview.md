## 一、基础与数据类型

### Redis 是什么、用在哪？（开胃）

**答：** **Redis**（Remote Dictionary Server）是 **内存** 为主的 **键值数据库**，也可做 **缓存、消息中间件、协调服务**。特点是 **丰富数据类型**、**高性能**、可选 **持久化** 与 **内置高可用（复制 / Sentinel / Cluster）**。与 **Memcached** 相比：Redis **结构更多、可持久化、Lua/事务、主从集群**；Memcached **多线程**、协议与模型更简单。**默认端口 6379**；客户端与服务端多用 **RESP**（REdis Serialization Protocol）编码命令与回复。

---

### 1. Redis 为什么快？（高频综合）

**答（层次化）：**

1. **内存访问**：工作集在 RAM，无磁盘寻道；瓶颈常在 **网络 RTT** 与 **序列化**，而非单次指令的 CPU。
2. **单线程执行命令**（经典模型）：**6.0+** 引入 **IO 多线程** 只做 **读取请求、写回响应**；**命令解析与执行** 仍在 **主线程**，避免 **多线程锁竞争** 和上下文切换（对比数据库多线程锁）。
3. **高效内核**：**epoll/kqueue** 等 **IO 多路复用** + **非阻塞** 事件循环，单进程可服务大量连接。
4. **数据结构为场景优化**：SDS、跳表、压缩列表/listpack、整数集合等，**时间复杂度** 可预期。
5. **纯内存 ≠ 永不阻塞**：**fork**（RDB、AOF 重写）、**大 key 删除/聚合**、**慢命令** 仍会 **卡住主线程**；面试可主动补这一点显得完整。

---

### 2. 常用数据类型及典型场景？

**答：** 先记 **原生五类**：String、Hash、List、Set、Sorted Set（ZSet）；再记 **扩展**：Stream、GEO、Bitmap、HyperLogLog、Bitfield 等。

| 类型 | 典型场景 | 常用命令（回忆用） |
|------|----------|-------------------|
| **String** | 缓存 JSON/二进制、计数、`INCR`、短锁 | `GET/SET`、`SET NX EX`、`INCR`、`MGET` |
| **Hash** | 对象字段、购物车SKU属性 | `HGET/HSET`、`HMGET`、`HINCRBY` |
| **List** | 队列/栈、时间线、阻塞队列 | `LPUSH/RPOP`、`BLPOP`、`LRANGE` |
| **Set** | 去重、标签、交并差（共同好友） | `SADD/SREM`、`SINTER`、`SUNION` |
| **ZSet** | 排行榜、延时队列、带权重的有序集合 | `ZADD`、`ZRANK/ZREVRANK`、`ZRANGEBYSCORE` |
| **Stream** | 持久消息流、消费组 | `XADD`、`XREADGROUP`、`XACK` |
| **Bitmap** | 签到、用户日活位图 | `SETBIT`、`BITCOUNT`、布隆可自建 |
| **HyperLogLog** | **UV 估算**，极小内存 | `PFADD`、`PFCOUNT`、`PFMERGE` |
| **GEO** | 附近的人（底层 ZSet + geohash） | `GEOADD`、`GEORADIUS`（或 `GEOSEARCH`） |

**追问：** **ZSet 的 member 须唯一**；相同 member 再次 `ZADD` 会 **更新 score**。**List 允许重复**，按插入顺序。**Set 无序、元素唯一**。

---

### 3. `String` 底层一定是字符串吗？

**答：** 逻辑上对使用者是「字符串/字节序列」，物理上经 **Redis Object** 指向 **SDS**（Simple Dynamic String）。**编码优化**（可用 **`OBJECT ENCODING key`** 查看）：

- **int**：值在 **64 位有符号整数** 可表示且字符串长合适时，**直接嵌在对象里**，O(1) 数学运算。
- **embstr**：**短字符串** 且 **只读多** 时，**对象头与 SDS 连续分配**，减少一次指针与缓存 miss（具体长度阈值随版本变化）。
- **raw**：**长字符串** 或不满足 embstr 条件，** SDS 单独分配**。

**SDS 相对 C 字符串**：**O(1) 取长度**、**预分配减少重分配**、**二进制安全（不以 \\0 结尾为界，可存图片/序列化数据）**、**惰性释放与拼接 API**。**面试主线：** SDS + 三编码 + `OBJECT ENCODING`。

---

### 4. `ZSet` 底层是什么？

**答：** **跳表（skiplist） + 哈希表（dict）** 组合：

- **跳表**：按 **score** 排序（score 相同则 **member 字典序**），支持 **范围查询**、**排名**，插入/删除/查找平均 **O(log N)**。
- **哈希表**：**member → score**，**O(1)** 查某个 member 的分数。

**为何常用跳表而非平衡树？** 实现简单、范围遍历友好、并发扩展历史上更省事（Redis 单线程执行命令，更关注 **实现复杂度与常数因子**）。**追问：** 「有序」还可用 **压测 + 索引** 在其它系统实现，Redis 选型是工程权衡。

---

### 5. `Hash` 何时用紧凑结构 / 哈希表？（方向）

**答：** 小对象用 **listpack**（Redis 7+ 主流，早期版本为 **ziplist**）**省内存**；字段数或单 value 长度超阈值则升级为 **hashtable** 换 **访问性能**。相关配置大致为（名称以当前版本 `redis.conf` 为准）：

- **`hash-max-listpack-entries`**：字段个数上限。
- **`hash-max-listpack-value`**：单个 value 最大字节数。

**升级特点：** 通常 **单向**：紧凑 → 哈希；**不会自动缩回**（需业务重建 key 才降体积）。**大 Hash** 仍是 **大 key** 来源，需 **分桶、拆分 field**。

---

### （基础补充）List 底层与复杂度

**答：** **Redis 3.2+** List 多为 **quicklist**：**双向链表** 连接多个 **listpack**（旧版为 **ziplist**）**节点**，兼顾 **插入删除** 与 **空间局部性**。两头 **O(1)**  `LPUSH/RPOP/LPOP`；**中间定位** `LINDEX` 约 **O(N)**（节点内可二分但需遍历节点）；**`LRANGE`** 一般按范围遍历。**阻塞读** `BLPOP` 适合 **简单任务队列**；**要求强持久化/堆积治理** 时常与 **Kafka/RocketMQ** 对比。

---

### （基础补充）Set 底层

**答：** 元素全为 **整数** 且数量小于阈值 → **intset**（有序整数数组 + 二分查找）；否则 **hashtable**。**`SINTER/SUNION/DIFF`** 复杂度与 **集合大小** 相关，**超大集合** 面经会问 **避免在线上对大 Set 做重交集** 或 **预先分片/近似**。

---

### （基础补充）Stream 要点（与 List/PubSub 区分）

**答：** **Stream** 是 **持久化** 的只追加日志结构；每条消息是 **`{ms}-{seq}`** ID。支持 **消费组（Consumer Group）**：`XGROUP CREATE`、`XREADGROUP` 拉取、`XACK` 确认、**PEL（pending list）** 记录 **已投递未 ACK**。**对比：** **List** 可被多个消费者 **竞争消费** 但无内建 **ACK/重投语义**；**Pub/Sub** **不存消息**，离线即丢。**Cluster** 下单 Stream **同普通 key** 占一个 **槽**，超大流需 **分 key / 分片**。

---

### （基础补充）Bitmap、HyperLogLog、GEO、Bitfield

**答：**

- **Bitmap**：`SETBIT key offset bit`，offset 很大时 **单 key 占内存** 上升，属 **大 key 风险**；可用 **分片 key**（按用户段、按天）。
- **HyperLogLog**：标准误差约 **0.81%**，**PFADD** 可去重计数，**不能列举元素**。
- **GEO**：经纬度 **geohash 编码** 后作为 **ZSet score 相关排序**，范围查询用 **`GEOSEARCH`**（新）或旧 **`GEORADIUS`**（视版本）。
- **Bitfield**：按 **位域** 存整型、自增等，适合 **紧凑状态位**；与纯 Bitmap 使用场景不同。

---

## 二、内存与淘汰

### 6. 内存达到 `maxmemory` 后怎么办？

**答：** **`maxmemory`** 设定 Redis **可用内存上限**（不含 AOF 缓冲区、复制缓冲等 **额外开销**，实际要留余量）。达到上限后的行为由 **`maxmemory-policy`** 决定：

| 策略（常见名） | 含义 |
|----------------|------|
| **noeviction** | **默认常见**：写命令 **拒绝**（返回错误），读仍可；适合 **不允许丢数据** 的侧车/队列场景。 |
| **allkeys-lru / allkeys-lfu** | 从 **全部 key** 中按 **近似 LRU 或 LFU** 淘汰；**纯缓存** 最常用。 |
| **volatile-lru / volatile-lfu** | 只从 **带 TTL** 的 key 中淘汰。 |
| **allkeys-random / volatile-random** | 随机淘汰，**较少** 作首选。 |
| **volatile-ttl** | 优先淘汰 **TTL 更短** 的 key。 |

**与「过期」区别：** **TTL 到期** 走 **过期删除**（见下）；**淘汰** 是 **内存压力** 下 **提前扔数据**。**缓存** 通常 **allkeys-lru/lfu + 合理 TTL**；**队列** 若不能丢消息 常 **noeviction** 并监控内存。

---

### 7. Redis 的 LRU 是严格 LRU 吗？

**答：** **不是**。实现为 **近似 LRU**：

- 每个 key 带 **24bit「时钟」** 记录 **最近一次访问时间**（精度有限）。
- **采样** 一批候选 key，在 **淘汰池** 里选 **最久未访问**；用 **可接受的误差** 换 **O(N)** 全表扫描的代价。

**LFU**（4.0+）：按 **访问频次** 淘汰；带 **衰减因子** 让 **老热点** 随时间降温，**对数计数** 节省空间。**追问：** 若面经问「哪种 policy 适合热点随时间漂移」→ 常答 **LFU 或 LRU** 要看「新热点 vs 长期稳定热点」。

---

### 8. `lazyfree` 是干什么的？

**答：** **惰性释放 / 异步删除**：对 **大对象**（大 Hash/List 等），**主线程** 若同步回收内存（`DEL`）会 **长时间阻塞**。**`lazyfree-lazy-eviction`**、**`lazyfree-lazy-expire`**、**`lazyfree-lazy-server-del`** 等开关控制是否在 **后台线程** 释放。**`UNLINK`** 语义上类似 **DEL**，但常走 **异步**（具体受配置与版本影响）。**运维** 删大 key 优先 **`UNLINK`** 或 **分批删**。**`FLUSHALL/FLUSHDB ASYNC`** 同理减轻尖刺。

---

### （基础补充）过期键如何真正「删掉」？

**答：** **两种机制配合**（都可能 **滞后**，故存在「已到 TTL 但仍暂占内存」的瞬间）：

1. **惰性删除**：**读/写命中** 某 key 时顺带检查是否过期，过期则 **删除**。**不占 CPU 扫描**，但 **长期不访问的冷 key** 可能残留。
2. **定期删除**：按 **hz** 配置的节奏，**间隔抽样** 一批过期 key **删除**，防止冷 key 堆积。

**不会** 为每个 key 起定时器（海量 key 不可行）。**Pub/Sub 的 keyspace 通知** 可与过期事件结合，但 **事件可能丢失**，不能当 **强一致定时器**。**面试收口：** 惰性 + 定期；与 **淘汰策略**、**持久化是否过期后仍可见** 区分（RDB：主保存时已过期的通常不写；AOF：重写时用内存视图「折叠」）。

## 三、持久化

### 9. RDB 与 AOF 区别？

**答：**

| | RDB | AOF |
|---|-----|-----|
| **方式** | 某时刻 **快照**（fork 子进程） | **追加写命令** |
| **恢复** | 快 | 慢（重放）；**AOF 重写** 压缩体积 |
| **丢失** | 两次快照间可能丢 | **everysec** 最多约 1s；**always** 更安全更慢 |

**两者都开**：重启 **优先用 AOF**（配置与版本默认策略以文档为准）。

**触发与配置（面经追问）：** **RDB** 可由 **`save m n`**（m 秒内至少 n 次写触发 **`BGSAVE`**）、手动 **`SAVE/BGSAVE`**；**AOF** 在 **`appendonly yes`** 下将写命令追加到文件。**`appendfsync`**：**always**（每条写刷盘，最安全最慢）、**everysec**（每秒刷，折中）、**no**（交 OS，快但丢窗口大）。

---

### 10. AOF 重写（rewrite）是什么？

**答：** **fork** 子进程 **基于内存当前数据** 写新 AOF，**合并命令** 等价恢复，**缩短文件**；**重写期间** 新写入进 **AOF 缓冲区**，**结束后追加**。

---

### 11. 写时复制（COW）与 `fork` 阻塞？

**答：** **`BGSAVE` / AOF 重写** 用 **fork**；**大实例** **fork 耗时长** 导致 **延迟尖刺**；**Linux 透明大页** 可能加剧。**持久化放从节点**、**磁盘 SSD**、**控制内存** 是常见手段。

**COW 简述：** fork 后父子 **共享物理页**；**任一方写** 时 **复制该页** 再修改，故 **快照/重写期间** 写入越猛 **拷贝页越多**、**RSS 与延迟** 越容易尖刺。

---

### （基础补充）混合持久化（AOF 里夹 RDB）？

**答：** **Redis 4.0+** 可开 **`aof-use-rdb-preamble yes`**：AOF **重写** 生成的新文件以 **RDB 格式快照** 开头，其后 **追加增量 AOF 命令**。**恢复** 时先加载 RDB 段再重放尾部 AOF，**折中重启速度与 AOF 连续性**（以当前版本文档为准）。

---

## 四、主从复制

### 12. 主从复制过程口头描述？

**答：** **从库** 向主发起 **`PSYNC` / `PSYNC2`**（版本相关）：**全量复制** 时主 **`BGSAVE`** 生成 **RDB** 发给从；**生成与传输窗口内** 主的新写入进入 **replication buffer**（与 **复制积压缓冲区 repl_backlog** 不同：前者保证本次全量期间的命令不丢）。从 **load RDB** 完成后，主再把缓冲区里命令 **追齐**。**增量** 若从的 **复制偏移** 仍落在 **`repl_backlog`** 环形缓冲区里，则 **部分重同步**；否则 backlog 被覆盖则 **再次全量**。**之后** 进入 **长连接命令传播**（异步）。**主宕** 自动切主靠 **Sentinel/Cluster**，手动则 **切从为新主** 并 **改拓扑**。

---

### 13. 复制延迟、数据不一致？

**答：** **异步复制** 必然 **主从延迟**；**读写分离** 读从可能 **旧数据**。**Wait** 命令（版本支持）可 **同步等待** N 个副本（**有代价**）。

---

## 五、Sentinel 与 Cluster

### 14. Sentinel 做什么？

**答：** **监控**、**故障转移**（选主、通知客户端）、**配置提供**。**多数 Sentinel 同意** 才 **failover**，避免 **脑裂误切**。

---

### 15. Redis Cluster 如何分片？

**答：** **16384 哈希槽**；**key** 对 **`{tag}`** 做 **CRC16 % 16384**（**hash tag** 控制同槽）；**MOVED**（永久重定向）、**ASK**（迁移中一次性）。

---

### 16. Cluster 限制常被问什么？

**答：** **多 key 操作** 需 **同槽**；**Lua** 访问 **多 key** 同限制；**无单机** 那么强的 **多 DB**（默认 **db0**）。

---

## 六、事务、Pipeline、Lua

### 17. Redis 事务是 ACID 吗？

**答：** **`MULTI/EXEC`**：**命令入队**，**EXEC 时一次性执行**；**无回滚**（某命令失败 **后续仍执行**）；**隔离性** 弱（**无乐观锁除非 WATCH**）。**不等同** 关系型事务。

---

### 18. `WATCH` 做什么？

**答：** **乐观锁**：**监控 key**，**EXEC 前被改则 EXEC 失败**，可 **重试**。

---

### 19. Pipeline 与事务区别？

**答：** **Pipeline**：**批量发命令**、**减少 RTT**，**不保证原子**；**事务**：**EXEC 块内** **顺序执行**，**仍非** 全失败回滚。

---

### 20. 为什么复杂逻辑常用 Lua？

**答：** **服务端原子执行**（单命令模型）、**减少网络往返**；注意 **执行时间**（**慢脚本** 阻塞）、**Cluster 多 key** 槽位。

---

## 七、缓存设计与经典问题

### 21. Cache Aside（旁路缓存）流程？

**答：** **读**：先缓存，miss 读 DB **再写回缓存。** **写**：**先更新 DB，再删缓存**（或 **先删再写** 各有 **脏读** 讨论，常见 **延迟双删** 缓解）。**面试讲清** 与 **并发** 顺序。

---

### 22. 缓存穿透？

**答：** 查询 **不存在的数据**，**绕过缓存** 打穿 DB。**解决**：**布隆过滤器**、**缓存空值**（**短 TTL**）、**接口校验**。

---

### 23. 缓存击穿？

**答：** **热点 key 过期**，瞬时 **大量请求打 DB**。**解决**：**互斥锁**（单线程回源）、**逻辑过期**、**热点永不过期 + 异步刷新**。

---

### 24. 缓存雪崩？

**答：** **大量 key 同时过期** 或 **Redis 宕机**。**解决**：**过期时间加随机**、**多级缓存**、**集群高可用**、**限流降级**、**熔断**。

---

### 25. 双写一致性怎么保证？（开放）

**答：** **强一致** 难：**消息队列**、**Canal 订阅 binlog** 更新缓存、**版本号**；**接受最终一致** + **TTL** + **幂等** 更常见。

---

## 八、性能与大 key / 热 key

### 26. 如何发现大 key、热 key？

**答：** **`redis-cli --bigkeys` / `--hotkeys`（采样）**、**SCAN + DEBUG OBJECT**、**监控** `latency`、`slowlog`、**客户端埋点**；**生产** 用 **云平台工具** 或 **代理层统计**。

---

### 27. 大 key 问题与治理？

**答：** **删除阻塞**（**UNLINK**）、**主从同步慢**、**内存不均**。**拆 key**、**压缩结构**、**异步删除**。

---

### 28. 热 key 问题与治理？

**答：** **单分片 CPU 打满**（Cluster）；**本地缓存**、**随机后缀多副本**（读扩散）、**读写分离**。

---

## 九、分布式锁与其它模式

### 29. `SET key value NX EX` 实现锁注意什么？

**答：** **过期时间** 防死锁；**value 用唯一 token**；**释放用 Lua 校验持有者**；**续期**（看门狗）；**仍非** 与 **ZK/etcd** 同级的 **强一致**，**主从异步** 下 **主宕机可能丢锁**（**Redlock** 争议点）。

---

### 30. Redlock 是什么？争议点？

**答：** **多独立 Redis 节点** 多数派加锁；**Antirez vs Martin Kleppmann** 论战：**时钟跳跃**、**GC 停顿**、**必要性**。**面试答**：**业务可接受风险** 与 **etcd/Consensus** 对比。

---

## 十、实战场景题

### 31. 场景：接口被刷，数据库扛不住？

**答：** 目标是在 **接入层或网关之后、打到 DB 之前** 把无效与过量请求压下去，**Redis 常做限流与计数**，但不是唯一手段。

**（1）限流**  
对 **用户 ID、IP、设备指纹** 等维度做 **令牌桶或漏桶**；**Redis + Lua** 保证 **原子自增与过期**，实现 **QPS 上限**。**单实例 Redis** 扛不住时再 **分片集群** 或把 **限流逻辑下沉到 API 网关**。

**（2）过滤无效请求**  
对 **明显不可能存在的 id** 用 **布隆过滤器**（RedisBloom 或本地布隆 + 少量 Redis），**穿透** 到 DB 前先挡一层。**缓存穿透** 还要对 **合法但不存在的数据** 做 **空值缓存**（短 TTL）。

**（3）业务与运维**  
**活动页、秒杀** 要 **页面静态化、CDN**；**WAF** 封 **恶意 IP**；**验证码、登录态** 提高刷接口成本。  
**收束：** **限流 + 过滤 + 缓存空对象 + 网关/WAF**，DB 只做 **该做的查询**。

---

### 32. 场景：秒杀库存怎么扣？

**答：** 核心是 **「不能超卖」** 与 **「扛住峰值」** 往往拆成 **Redis 预减 + 异步落库** 两阶段。

**（1）典型路径**  
**Redis** 用 **`DECR` / Lua 脚本** 在 **单 key** 上 **原子扣减**，扣减成功即 **获得下单资格**，再发 **MQ** 异步创建订单；**DB 最终扣库存** 时用 **`UPDATE ... WHERE qty >= ?`** 或 **乐观锁** 再 **兜底防超卖**。  
**Lua** 可以一次判断 **库存、用户是否重复抢、活动是否结束**，避免多次往返。

**（2）注意**  
**Redis 与 DB 不一致** 时要 **对账、补偿、超时关单释放库存**；**热点 key** 可能 **单分片瓶颈**，需 **拆分 key 或分桶库存**（业务上可接受时）。**不能只依赖 Redis DECR** 当 **最终库存**，除非业务允许 **强最终一致 + 对账**。

---

### 33. 场景：延时任务（30 分钟后关单）？

**答：** Redis 适合做 **轻量延时任务**，复杂调度仍建议 **专业队列**。

**（1）ZSet 方案**  
**member** 为 **订单 ID** 等业务键，**score** 为 **到期时间戳**；**定时任务**（如每秒）用 **`ZRANGEBYSCORE` 拉取到期项**，**Lua 消费并删除**，再触发 **关单逻辑**。注意 **多实例** 时 **应用层分布式锁** 或 **单线程扫描** 避免重复执行。

**（2）Keyspace Notifications**  
**键过期** 通知 **不可靠**（可能丢失、延迟），**不适合** 强金融关单，仅作**辅助**。

**（3）更重场景**  
**RocketMQ 延时消息、定时调度平台（XXL-JOB 等）** 提供 **可观测、重试、堆积**；面试可对比 **Redis 轻量 vs MQ 完备**。

---

### 34. 场景：排行榜 Top100？

**答：** **ZSet** 是标准答案：**score 为积分/战力**，**member 为用户 ID**；**`ZREVRANGE 0 99 WITHSCORES`** 取 Top100，**O(log N)** 插入。

**（1）数据量大时**  
**全站用户** 一个 ZSet 过大时，可按 **赛季、分区、活动** 拆 **多个 key**；或 **只维护 Top K**（定期裁剪低分）。**日榜/周榜** 用 **不同 key + TTL**。

**（2）持久化**  
**RDB/AOF** 与 **过期策略** 要评估：**排行榜是否允许丢**（可丢可缩短持久化）；**核心资产** 可 **异步写 DB 备份**。

**（3）超大规模**  
**近似 TopK**（如 **HeavyKeeper** 等）、**分桶聚合** 是扩展方向，**面试提一句即可**。

---

### 35. 场景：UV 统计，内存要省？

**答：** **UV 是去重计数**，若用 **Set** 存每个用户 id，**内存随 UV 线性涨**。

**（1）可接受误差**  
**HyperLogLog**：`PFADD` 用户标识，`PFCOUNT` 得到 **估算 UV**，**内存固定很小**（几 KB 到十几 MB 量级，与配置有关），**误差约 0.81%**。

**（2）要精确**  
**Bitmap**：用户 id 可映射为 **连续整数** 时极省；**稀疏** 可用 **Roaring 或分片 Bitmap**。**Set** 仍适合 **小规模活动**。

**（3）面试收束**  
先问 **是否允许误差** → **HLL vs Bitmap vs Set**；**大促 UV** 常见 **HLL + 分桶**。

---

### 36. 场景：分布式 Session？

**答：** 多实例下 **Session 不能存本地**，**Redis** 存 **Session** 是常见方案。

**（1）结构**  
**Hash**：`HSET session:{id} field value`，**TTL** 与 **滑动过期**（每次访问续期）按业务选。**Spring Session** 封装了 **Redis 序列化** 与 **Cookie**。

**（2）登出与安全**  
**登出** 时 **`DEL session:{id}`**；**敏感字段** 不要塞进 Session；**序列化** 变更（JSON 版本、类名）要考虑 **兼容**。

**（3）风险**  
**Redis 宕机** 会 **全员登出**，可 **主从/集群**；**会话劫持** 靠 **HTTPS、HttpOnly Cookie**。

---

### 37. 场景：限流：滑动窗口？

**答：** **固定窗口** 在边界处可能 **突刺**（两倍流量）；**滑动窗口** 用 **时间窗口内请求数** 更平滑。

**（1）Redis 实现**  
**ZSET**：`member` 为 **毫秒时间戳+随机串** 保证唯一，`score` 为 **时间戳**；每次请求 **`ZADD` + `ZREMRANGEBYSCORE` 删掉窗口外 + `ZCARD` 与阈值比较`**，可用 **Lua 原子化**。  
**或** **Redis Cell 模块**（若部署支持）、**令牌桶 Lua**。

**（2）对比**  
**滑动窗口** 实现与 **存储** 成本高于 **固定窗口 + 漏桶**；**高并发** 可 **本地限流 + Redis 全局限流** 两层。

---

### 38. 场景：主从切换后锁丢失？

**答：** **根因** 是 **主从异步复制**：客户端在 **Master** 上 **`SET NX` 成功** 拿到锁，**尚未同步到 Slave** 时 Master 宕机，**Slave 升主** 上没有这把锁，**另一客户端** 也能加锁 → **双持锁**。

**（1）面试结论**  
**单 Redis 主从** 不是 **强一致分布式锁**；**Redlock**（多独立节点多数派）仍被质疑 **时钟与 GC**（见 **题 30**）。**金融级互斥** 往往选 **etcd**、**ZooKeeper** 或 **DB 悲观锁**。

**（2）工程上**  
若业务 **容忍极小概率双持**（例如 **防重入幂等**），可 **Redis 锁 + 过期 + 续期**；否则 **不要用 Redis 当唯一真相**。

---

### 39. 场景：缓存与 DB 同时更新谁先谁后？

**答：** 经典 **Cache Aside** 变体：**优先保证 DB 为真相**。

**（1）推荐路径**  
**先更新 DB，再删缓存**（或 **设置新值带版本**）。**读** 时 **miss 再加载 DB 回写**。**「先删缓存再更新 DB」** 在并发下 **旧值写回缓存** 风险更大。

**（2）延迟双删**  
**更新 DB 后** **延迟再删一次缓存**，降低 **并发读写** 窗口内的脏数据概率；**延迟时间** 要 **大于主从延迟**（若有）。

**（3）更稳**  
**订阅 binlog（Canal 等）异步删缓存**，**对缓存与 DB 解耦**；**版本号** 写入缓存，**读时对比**。

---

### 40. 场景：Redis 内存暴涨如何应急？

**答：** **先止血再定位**，避免 **生产 KEYS**。

**（1）定位**  
**`SCAN` + `MEMORY USAGE`**、**`redis-cli --bigkeys`**（采样）、**监控** `used_memory`、`maxmemory`；看 **热点 key**、**大 Hash/List**、**客户端连接数**。

**（2）临时**  
**调 `maxmemory-policy`**（如 **volatile-lru** 需有过期 key）；**业务侧** 删 **可废弃** 的大 key；**扩容**（垂直或 **Cluster 分片**）。

**（3）长期**  
**拆分大 key**、**合理 TTL**、**压缩 value**、**冷热分离**；**禁止** `KEYS` 与 **无界集合**。

## 十一、进阶补充

### 41. `KEYS` 为何生产禁用？替代？

**答：** **`KEYS` O(N) 阻塞** 主线程。**`SCAN`** **游标迭代**，**每次少量**，**可能重复** 需客户端去重，**适合** 巡检。

---

### 42. Redis 6+ ACL 做什么？

**答：** **用户、密码、命令子集、 key 模式** 权限；**替代** 弱口令 **全权限**；**最小权限** 原则。

---

### 43. Client Side Caching（客户端缓存）要点？

**答：** **6.0+** **跟踪 key**，**失效** 时 **服务端推送** **无效消息**；**降低** 热点读 **网络**；**一致性** 为 **最终一致**，**需** 客户端 SDK 支持。

---

### 44. `MEMORY DOCTOR` / `MEMORY USAGE`？

**答：** **诊断** 内存碎片、**建议**；**单 key 内存** 估算；**配合** **分析大 key**。

---

### 45. Redis Functions（7+）与 Lua？

**答：** **持久化到库** 的 **函数**（**可备份复制**），**比** 客户端散落 **Lua** **易治理**；**热更新** 与 **权限** 更清晰（**方向性** 描述）。

---

### 46. 模块化生态（面试加分）？

**答：** **RedisBloom**（布隆）、**RedisSearch**、**RedisJSON**、**RedisTimeSeries** 等；**解决** 纯原生类型 **组合复杂** 的问题；**部署** 需 **Redis Stack** 或 **自建模块**。

## 十二、面经普通题补充（近年）

> 与上文 **题 1～46** 互补；覆盖面经中 **反复追问** 的对比题、口诀题与 **八股扩展**。

### 47. Redis 与 Memcached 区别？（面经对比）

**答：** **Redis**：**丰富数据结构**（Hash、ZSet、Stream 等）、**持久化（RDB/AOF）**、**主从/哨兵/集群**、**Lua**。**Memcached**：**纯 KV**、**多线程**、**简单缓存**、一般 **无持久化**。选型：**复杂缓存、排行榜、限流、分布式锁** 倾向 Redis；**纯简单 KV、极高 QPS 简单场景** 历史上 Memcached 仍有，但 **Redis 生态** 更全。

### 48. 缓存穿透、击穿、雪崩完整答法？（面经八股）

**答：** **穿透**：查 **不存在的数据**，缓存不起作用，打穿到 DB → **布隆过滤器**、**缓存空值（短 TTL）**、**接口校验**。  
**击穿**：**热点 key 过期** 瞬间大量请求打 DB → **互斥锁重建**、**逻辑过期**、**热点永不过期 + 异步刷新**。  
**雪崩**：**大量 key 同时过期** 或 **Redis 宕机** → **随机 TTL**、**多级缓存**、**集群高可用**、**熔断降级**。  
与 **题 39**（先 DB 后删缓存）区分：**这是另一类「经典三问题」**。

### 49. Pipeline 与事务（MULTI/EXEC）区别？

**答：** **Pipeline**：**一批命令一次 RTT** 发过去，**不保证原子性**，只是 **减少网络往返**。**事务**：**MULTI/EXEC** 将命令 **排队一次执行**，**无回滚**；**WATCH** 实现 **乐观锁**。**Lua**：**单脚本原子** 且 **逻辑复杂** 时常用。面经：**Pipeline 提速，Lua 原子**。

### 50. 热 key / 大 key 如何发现与处理？

**答：** **发现：** **`--hotkeys`**（需 **LFU** 等前提）、**监控** `INFO`、**客户端埋点**、**Redis 6+ ACL 审计**。**大 key：** **`SCAN` + `MEMORY USAGE`**、**`redis-cli --bigkeys`**。  
**处理：** **热 key** → **本地缓存、拆分 key、读写分离**；**大 key** → **拆分 Hash/List**、**异步删除 `UNLINK`**。与 **题 40** 呼应。

### 51. 共同好友 / 交集类需求怎么用 Redis？

**答：** 用户关注集合用 **Set** 存；**共同关注** **`SINTER user:A user:B`**；**可能认识的人** **`SDIFF`** 等。**时间复杂度** 与 **集合大小** 有关，**超大 Set** 需 **分片或近似**。

### 52. Redisson「看门狗」面经里指什么？

**答：** **分布式锁** 过期时间到了但 **业务未执行完**，**看门狗线程** 自动 **续期** `TTL`，减少 **误释放锁**。仍受 **主从异步** 与 **Redlock 争议** 约束；面试 **提一句即可**，深问见 **题 29～30、38**。

### 53. Redis 单线程为何还能高并发？（面经追问）

**答：** **命令执行** 主线程 **单线程** 避免 **锁**；**瓶颈** 往往在 **内存访问与网络**，不在 **CPU 多核**。**6.0+** **IO 线程** 分担 **网络读写**，**命令仍主线程**。配合 **多实例/集群** 用满多核。
